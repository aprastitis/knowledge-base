# Agentic Coding Principles & Practices

## Type
Engineering Practice — AI Collaboration Framework

## Summary
Six principles and twenty-eight practices for moving beyond "vibe coding" with AI agents — from quick demos to production-quality, accountable software development. The core shift: treat AI coding agents as powerful but needing direction, not as autonomous black boxes that produce reliable results by default.

## Origin
Created by **0xdepressionn** (Twitter/X). Inspired by Ethan Mollick's "Speaking things into existence" concept. Shared at [agentic-coding.github.io](https://agentic-coding.github.io/).

## The Core Problem

AI coding agents can instantly turn ideas into code — almost like "speaking things into existence." Like casting a fireball with a simple incantation. But:

- The results are often messy, hard to control
- Code that works for a demo may be inadequate for production
- "Technical debt" accumulates invisibly until it becomes critical

**What we need:** Not fireballs, but meteor strikes — powerful, sophisticated, reliable outcomes. Getting there requires the developer (the "caster") to have deep understanding, a cautious approach, and skilled "spell control."

## Six Principles

### 1. Developer Accountability
Even when assisted by an AI agent, the developer holds ultimate responsibility for quality, functionality, performance, security, and maintainability of the code.

**Core:** AI is merely a tool; the human is the one accountable. "The AI did it" is not a valid excuse.

### 2. Understand and Verify
Code generated or suggested by AI should not be accepted or used without clear understanding. Before integrating any code, its functionality and implications must be fully understood and thoroughly verified.

**Core:** Blindly accepting code is prohibited. This prevents hidden bugs or unintended side effects.

### 3. Prioritize Security and Confidentiality
Do not input or share sensitive company information (source code, API keys, internal data, customer information, IP) directly with external AI agents not explicitly approved by the company. Remain vigilant about unintentional data leakage.

**Core:** Prevent data breaches and security incidents.

### 4. Maintain Code Quality, Standards, Consistency
Any code generated by an AI agent must be modified and managed to meet the team's coding conventions, architectural patterns, design systems, security standards, and compliance requirements.

**Core:** Maintain codebase consistency and quality; manage technical debt actively.

### 5. Human-Led Design and Critical Thinking
Core system design, architectural decisions, and critical business logic must be human-led. Use AI agents in a supporting role (ideation, implementation assistance), but always critically evaluate their suggestions.

**Core:** Recognize AI's limitations; base crucial decisions on human insight and experience.

### 6. Recognize AI's Limitations and Adapt to Technological Change
Be aware of AI's capabilities and recognized limitations: hallucinations, biases, outdated knowledge, lack of contextual understanding. Keep pace with evolving AI technologies and tools.

**Core:** Do not over-rely on AI; clearly understand its boundaries. Continuously adapt and build team capabilities.

## Twenty-Eight Practices

### A. Preparation and Setup

**1. Set Agent Rules and Base Context**
Actively leverage custom configuration features (project-specific agent instructions, persistent context files) to preset rules the AI must always adhere to, along with persistent contextual information. Include: team coding standards summary, architectural principles, key libraries/versions, GitFlow policies, tech stack conventions.

> **For Hermes:** Use system prompt customization, workspace instructions, and project context to define rules and baseline context for each project.

**2. Provide Project Structure and Design Context**
For an AI agent to correctly modify or generate code within a specific project, provide explicit structural and design context: roles and responsibilities of key directories/modules/classes, data flow patterns, library usage conventions.

> **For Hermes:** When working on a project, ensure Hermes has clear context about directory structure, architecture decisions, and established patterns.

### B. Strategic AI Usage

**3. Define Task-Specific AI Strategies**
Before initiating a task, determine if it's one where AI can largely drive implementation (feature implementation, refactoring, test generation — based on detailed specs), or if it requires significant human involvement (architectural design, sensitive data logic, complex business rules).

**Core:** Match the AI's role to the task type. Provide highly detailed specs for AI-driven tasks. Ensure deep human involvement from design through review for complex tasks.

**4. Adapt AI Strategy Based on Context**
When AI repeatedly offers inaccurate code, suggests stopgap solutions, introduces security risks, or shows clear performance limitations — flexibly adjust your approach:
- Instruct the AI to use web search for analogous problem cases
- Try a different LLM if the current one isn't suitable
- Re-orient with broader context and guide to root cause analysis
- Request deeper thinking/planning for complex problems

**5. Leverage Multiple Agents (Parallel Use / Collaboration)**
Move beyond single-agent use:
- **Specializing agents:** one generates code, another reviews for quality/security
- **Role division:** one writes tests, another writes code to pass them
- **Parallel processing:** run different tasks concurrently in multiple agent instances

> **For Hermes:** Hermes's multi-agent Kanban board (`hermes_kanban`) is a natural fit for multi-agent collaboration patterns.

### C. Interaction and Prompting

**6. Craft Specific and Clear Prompts**
AI cannot perfectly infer human intent. Avoid vague requests. Formulate prompts that are specific and unambiguous — clearly state the goal (What), context (Where), constraints, and requirements (How). Iteratively refine if output doesn't meet expectations.

**7. Decompose Tasks into Manageable Units**
AI performs better on clear, manageable units of work rather than large, complex tasks. Decompose work into meaningful, smaller steps (functions, modules, small increments). Request these sequentially and integrate results. Find the appropriate level of decomposition based on task complexity and AI capabilities.

**8. Ensure Consistency with Code Examples (Few-Shot Prompting)**
AI generates more accurate results when provided with specific examples. When adding or modifying features, include relevant existing code snippets or examples of similar functionality. This guides the AI to learn style, structure, and patterns.

**9. Prioritize Exploration Before Implementation/Planning**
When tackling complex problems or unfamiliar codebases, instruct the AI to first explore relevant background information — read specified files, documentation, URLs — before generating code or creating a plan. Prevent premature solution generation during this phase.

**10. Adopt a "Plan First, Code Later" Approach**
For complex or critical tasks, require the AI to first outline its implementation plan or approach. Review and approve the plan before instructing the AI to generate code. Catches misunderstandings early.

**11. Monitor and Intervene During AI Task Execution**
Even with clear instructions, AI can head in unintended directions. If the agent exhibits unexpected behavior — performing unrequested tasks, modifying code when only asked a question, altering code unpredictably — immediately intervene and provide course correction.

### D. Code Quality and Review

**12. Review Every Change Before Committing**
AI-generated code must pass through human review before integration. Check for: correctness, security implications, maintainability, alignment with project conventions.

**13. Maintain a Test-First Mindset**
Use AI to generate tests, but ensure tests actually validate the intended behavior. Don't accept tests that just "pass" — verify they test what matters.

**14. Enforce Consistency with Linters and Formatters**
Configure project linters and formatters as non-negotiable guardrails. AI-generated code must pass the same quality gates as human-written code.

**15. Document the "Why", Not Just the "What"**
AI can explain what code does, but the intent and context behind decisions often gets lost. Ensure code comments capture the reasoning, not just the mechanics.

### E. Security and Compliance

**16. Never Blindly Trust AI with Security-Sensitive Code**
Code involving authentication, authorization, payment processing, or data handling requires extra scrutiny when generated by AI. Apply security review checklist.

**17. Keep Secrets and Credentials Out of AI Contexts**
Never paste API keys, tokens, passwords, or credentials into AI prompts or context. Use environment variables and secret management tools.

**18. Validate AI Suggestions Against OWASP Guidelines**
Cross-reference AI suggestions for input handling, SQL construction, and data serialization against OWASP security guidelines.

### F. Iterative Refinement and Learning

**19. Iterate on Prompts Based on Outcomes**
Track which prompt patterns produce good results and which don't. Maintain a running log of effective prompting strategies for your project.

**20. Give AI Feedback on Generated Code**
When AI-generated code is rejected or needs major rework, explain why. This helps the AI align better in subsequent interactions.

**21. Preserve Institutional Knowledge**
AI can forget or hallucinate project-specific patterns. Maintain durable documentation (ADR, architecture decision records, coding guides) that anchors institutional knowledge outside the AI's context window.

### G. Measurement and Accountability

**22. Define Clear Acceptance Criteria Before Starting**
Every task should have documented acceptance criteria before AI work begins. This prevents scope creep and provides a clear definition of "done."

**23. Track AI Contribution Separately**
When AI generates a significant portion of code, note this in commit messages or code comments. This surfaces AI's impact on the codebase over time.

**24. Conduct Regular AI Pairing Retrospectives**
Periodically review how AI assistance is affecting code quality, velocity, and defect rates. Adjust practices based on data.

### H. Advanced Patterns

**25. Use AI for Ideation and Explore Multiple Options**
Before committing to an approach, use AI to explore alternatives. Ask "what are three different ways to solve this?" and evaluate trade-offs.

**26. Leverage AI for Test Generation and Edge Case Discovery**
AI excels at generating edge case tests humans might miss. Use this systematically to harden critical code paths.

**27. Apply AI to Codebase Maintenance Tasks**
Refactoring, dead code elimination, documentation updates, dependency updates — AI can handle significant maintenance burden if properly directed.

**28. Build Team AI Literacy Incrementally**
Share effective patterns, failures, and learnings with the team. AI assistance quality improves as shared understanding grows.

## Key Insight: From Fireball to Meteor

The fundamental shift is from treating AI as an autonomous generator to treating it as a powerful tool that requires skilled direction:

| Fireball (vibe coding) | Meteor (agentic coding) |
|------------------------|-------------------------|
| Quick, impressive, hard to control | Powerful, precise, reliable |
| Works for demos and experiments | Suitable for production |
| High technical debt | Sustainable quality |
| AI is the decision-maker | Human is the accountable party |
| "Good enough" mindset | "Properly done" mindset |
